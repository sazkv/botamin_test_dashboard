# Botamin Analyst Dashboard

Botamin Analyst Dashboard - веб-дашборд для аналитика, который помогает быстро оценивать качество звонков AI-продажника, находить слабые места воронки и выбирать звонки для ручной проверки.

Дашборд сфокусирован на рабочих задачах аналитика: увидеть ключевые метрики, понять, где бот теряет клиентов, открыть нужную выборку звонков, проверить карточку звонка и исправить авторазметку.

## Стек

- React
- TypeScript
- Vite
- CSS без UI-фреймворка
- GitHub Pages для публикации статического frontend
- REST API поверх подготовленного backend и Postgres

## Продуктовая Логика

Дашборд разделяет звонки на несколько бизнес-категорий:

- все звонки
- неотвеченные звонки
- отвеченные звонки
- пустые дозвоны, автоответчики и тишина
- осмысленные разговоры
- разговоры, дошедшие до оффера
- разговоры, где предложена встреча
- разговоры, где встреча назначена
- звонки с проведенной квалификацией
- горячие лиды

Главная цель продукта - не просто показать список звонков, а помочь аналитику понять, на каком этапе воронки теряется клиент и какие звонки нужно разобрать вручную в первую очередь.

## Основные Разделы

### Dashboard

Главный экран для быстрой оценки проекта.

На экране отображаются:

- KPI-карточки по звонкам, ответам, осмысленным разговорам, встречам, квалификациям и горячим лидам
- интерактивная карта воронки с переходом в отфильтрованный список звонков
- блок `Где теряем клиентов` с этапами и причинами срыва
- блок `Очередь для аналитика` с приоритетными звонками для ручной проверки

Интерактивная воронка показывает движение звонков по цепочке:

```text
Всего звонков
-> Не отвечено
-> Отвечено
   -> Пустой дозвон / автоответчик / тишина
   -> Осмысленный разговор
      -> Приветствие пройдено
      -> Оффер раскрыт
      -> Встреча предложена
      -> Встреча назначена
      -> Квалификация проведена
      -> Горячий лид
```

Клик по любому узлу воронки открывает вкладку `Calls` с примененным фильтром.

### Calls

Рабочая таблица звонков для анализа и ручной проверки.

Доступные фильтры:

- период
- клиент
- компания
- статус
- отвеченные / неотвеченные
- только осмысленные разговоры
- этап воронки
- место срыва
- встреча назначена
- квалификация
- проверено аналитиком
- есть аудио

Таблица показывает:

- клиента
- дату и время звонка
- компанию
- нормализованный статус
- этап воронки
- место срыва
- признак назначенной встречи
- время созвона с экспертом
- квалификацию
- статус ручной проверки
- аудио
- ссылку на карточку звонка

В таблице используется `normalized_status`. Технический статус провайдера не используется как бизнес-статус и показывается только внутри карточки звонка.

## Карточка Звонка

Карточка открывается справа поверх таблицы и объединяет бизнес-информацию, технические поля, ручную проверку и transcript.

В карточке есть:

- клиент, дата, время и длительность звонка
- ссылка на аудио
- технический статус
- причина завершения
- итог звонка
- тип контакта
- максимальный этап воронки
- место и причина срыва
- AI summary и AI comment
- форма ручной проверки аналитика
- transcript репликами `user` и `bot`

Если transcript невозможно распарсить на реплики, он показывается plain text.

## Ручная Проверка

Аналитик может исправить авторазметку прямо в карточке звонка.

Поля ручной проверки:

- `qualification_is_correct`
- `manual_qualification`
- `manual_funnel_stage`
- `manual_failure_stage`
- `manual_failure_reason`
- `analyst_comment`

Доступные значения ручной квалификации:

- Не квалифицирован
- Холодный
- Тёплый
- Горячий
- Не целевой
- Нужен перезвон
- Назначена встреча
- Ошибка авторазметки

После сохранения frontend отправляет правки в backend endpoint ручной проверки.

## API

Frontend использует REST API с базовым URL из переменной `VITE_API_BASE_URL`.

Если переменная не задана, используется `/api`.

Используемые endpoints:

- `GET /api/calls` - список звонков
- `GET /api/calls/:id` - детальная карточка звонка
- `PATCH /api/calls/:id/analyst-review` - сохранение ручной проверки аналитика

Payload для сохранения ручной проверки:

```json
{
  "qualification_is_correct": true,
  "manual_qualification": "Горячий",
  "manual_funnel_stage": "Встреча назначена",
  "manual_failure_stage": "",
  "manual_failure_reason": "",
  "analyst_comment": "Комментарий аналитика"
}
```

Frontend нормализует входящие записи на клиенте и поддерживает несколько вариантов названий полей, чтобы работать с подготовленным API и промежуточными выгрузками.

## Правила Отображения Данных

- `company` остается пустым, если в данных нет компании.
- В таблице звонков показывается `normalized_status`.
- `technical_status` показывается только в карточке звонка как техническая информация.
- Пустые дозвоны и автоответчики не смешиваются с осмысленными разговорами.
- Ручные правки аналитика отображаются поверх автоматической разметки.

## Локальный Запуск

```bash
npm install
npm run dev
```

Пример `.env`:

```bash
VITE_API_BASE_URL=http://localhost:8000/api
```

Production build:

```bash
npm run build
```

Preview production build:

```bash
npm run preview
```

## Структура Проекта

```text
src/
  App.tsx        # основной UI: Dashboard, Calls, карточка звонка
  api.ts         # API client и нормализация записей звонков
  analytics.ts   # расчет KPI, воронки, потерь и очереди аналитика
  types.ts       # типы данных
  styles.css     # визуальный стиль dashboard
  main.tsx       # entrypoint React
```
